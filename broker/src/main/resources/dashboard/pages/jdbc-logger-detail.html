JSON Schema Configuration Guide

Overview

The JSON Schema defines how incoming MQTT messages are validated and mapped to database columns. It supports JSON Schema validation, JSONPath field mapping, array expansion, and automatic timestamp handling.

Schema Structure

The schema consists of four main parts:

Property Required Description
properties Yes Defines database columns with their data types
required No Array of field names that must be present
mapping No JSONPath expressions to extract values from incoming JSON
arrayPath No JSONPath to extract array for expanding into multiple rows

Data Types and Formats

Supported JSON Schema types:

  • string - Text values (VARCHAR)
  • number - Numeric values (INTEGER, FLOAT, DOUBLE)
  • boolean - True/false values
  • string with "format": "timestamp" - Timestamp fields that parse ISO 8601 strings or epoch ms (auto-fills current time if missing)
  • number with "format": "timestampms" - Timestamp fields that interpret the value as milliseconds since epoch (auto-fills current time if missing)

JSONPath Mapping

Use the mapping object to extract values from nested JSON structures using JSONPath expressions:

{ "properties": { "ts": {"type": "string", "format": "timestamp"}, "sensor_id": {"type": "string"}, "value": {"type": "number"} }, "mapping": { "ts": "$.timestamp", // Extract from root.timestamp "sensor_id": "$.device.id", // Extract from nested object "value": "$.readings.temperature" } }

Common JSONPath expressions:

  • $.field - Root level field
  • $.parent.child - Nested field
  • $.array[0] - First array element
  • $.sensors[*] - All array elements

Array Expansion

Use arrayPath to expand array items into multiple database rows. Root-level fields are merged with each array item:

Example: Expanding Sensor Array
Incoming JSON:
{ "device_id": "sensor-123", "timestamp": "2024-01-15T10:30:00Z", "sensors": [ {"id": "temp-1", "value": 23.5}, {"id": "temp-2", "value": 24.1}, {"id": "humidity-1", "value": 65.0} ] }
Schema:
{ "properties": { "ts": {"type": "string", "format": "timestamp"}, "device_id": {"type": "string"}, "sensor_id": {"type": "string"}, "value": {"type": "number"} }, "arrayPath": "$.sensors[*]", "mapping": { "ts": "$.timestamp", "device_id": "$.device_id", "sensor_id": "$.id", "value": "$.value" } }
Result: 3 database rows created
Row 1: {ts: '2024-01-15T10:30:00Z', device_id: 'sensor-123', sensor_id: 'temp-1', value: 23.5} Row 2: {ts: '2024-01-15T10:30:00Z', device_id: 'sensor-123', sensor_id: 'temp-2', value: 24.1} Row 3: {ts: '2024-01-15T10:30:00Z', device_id: 'sensor-123', sensor_id: 'humidity-1', value: 65.0}

Timestamp Handling

Two timestamp formats are available:

"format": "timestamp" - Flexible timestamp parsing:

  • Automatically parses ISO 8601 format: "2024-01-15T10:30:00Z"
  • Accepts epoch milliseconds as string or number: 1705318200000
  • If field is missing, uses current timestamp automatically

"format": "timestampms" - Milliseconds since epoch only:

  • Interprets long/number values as milliseconds since epoch: 1705318200000
  • Use this when you have numeric timestamps to avoid parsing overhead
  • If field is missing, uses current timestamp automatically

Complete Example

Full Configuration for IoT Sensor Data
Schema:
{ "type": "object", "properties": { "ts": { "type": "string", "format": "timestamp" }, "device_id": { "type": "string" }, "sensor_type": { "type": "string" }, "value": { "type": "number" }, "unit": { "type": "string" } }, "required": ["value"], "arrayPath": "$.sensors[*]", "mapping": { "ts": "$.timestamp", "device_id": "$.device_id", "sensor_type": "$.type", "value": "$.reading", "unit": "$.unit" } }
Database Table (must be created manually):
CREATE TABLE sensor_data ( ts TIMESTAMP, device_id VARCHAR, sensor_type VARCHAR, value DOUBLE, unit VARCHAR );

Important Notes

  • Database tables must be created manually before starting the logger
  • Column names in schema must match database column names exactly
  • If no mapping is provided, field names are used directly
  • Array expansion creates one row per array item
  • Fields not in schema are ignored
